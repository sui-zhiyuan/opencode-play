# 矩阵计算中的 Tiling 与 Tuning

## Tiling（分块 / 分片）

将大矩阵切分为能放进缓存的子块（tile），逐块计算，减少缓存替换（cache thrashing）。

### 动机

矩阵乘 `C[M×N] += A[M×K] × B[K×N]`，当 M、K、N 很大时，A、B、C 的总数据量远超缓存容量，导致每个元素被反复从主存加载。

```
朴素方式：遍历整个矩阵 → A、B、C 在缓存中互相踢出
→ 每个元素被多次从主存加载 → 带宽瓶颈
```

### 分块做法

将三个维度分别切块：

```
M → 多个 m_tile（如 64）
N → 多个 n_tile（如 64）
K → 多个 k_tile（如 256）

对于每个 (m_tile, n_tile, k_tile)：
  C_tile[m_tile][n_tile] += A_tile[m_tile][k_tile] × B_tile[k_tile][n_tile]
```

一个 tile 计算期间，A_tile、B_tile、C_tile **都留在 L1/L2 缓存**中，每个数据从主存只加载一次。

### 效果

```
未 tiling：A 的每个元素被重复加载约 N 次
已 tiling：A 的每个元素被加载约 ⌈K/k_tile⌉ 次
```

实际加速比通常在 5-20×，取决于矩阵大小和缓存层级。

### 缓存层级分块

现代库通常做 **多级 tiling**，对应 L1 / L2 / L3 的容量：

```
寄存器 tile（~32x4x4）      ← 一次内积展开，全部在寄存器中
L1 tile    （~256x256×?）   ← 数据在 L1 内
L2 tile    （~2K×2K×?）     ← 数据在 L2 内
L3/主存 block               ← 剩余的全局切分
```

## Tailing（尾处理）

当维度不能被向量宽度或 tile 大小整除时，处理余数部分。

### SIMD 尾处理

SVE 向量宽度 = 256 bits = 8 个 float。N = 30 → 3 次完整循环 + 余 6。

**SVE 方式（谓词掩码，天然支持）：**

```asm
    MOV     x0, #0
.Lloop:
    WHILELT p0.s, x0, x1           // 生成谓词：active lanes = x0 < x1
    LD1W    {z0.s}, p0/z, [x2, x0, LSL #2]
    FADD    z0.s, p0/m, z0.s, z1.s
    ST1W    {z0.s}, p0, [x3, x0, LSL #2]
    INCD    x0                      // x0 += 向量宽度（运行时可变）
    B.LT    .Lloop
```

`WHILELT` 自动生成谓词——最后一次迭代自动只激活剩余 6 个 lane，**不需要单独的尾处理分支**。

**NEON 方式（固定 128-bit，必须手动处理）：**

- 主循环：每次 4 个 float
- 尾处理：用 mask load/store 或标量循环补剩余的 1-3 个元素

### Tiling 尾处理

M = 200，m_tile = 64 → 3 个完整 tile（64×3 = 192）+ 尾 tile 大小 8。

```
完整 tile（size = 64）：常规最优 tiling 路径
尾 tile（size = 8） ：须用缩小参数或降级为标量
```

尾 tile 效率低但占比小（本例仅 4%），对总体性能影响有限。

### 微内核打包的对齐约束

tile 尺寸有时需要为微内核的展开因子（如 4、8）的整数倍，否则尾处理会影响打包缓冲区的布局，导致拷贝或掩码开销。常见做法是选择 **block 尺寸为展开因子的整数倍**，避免内层循环出现 tailing。

## Tuning（调优 / Auto-Tuning）

在给定硬件上，搜索最优参数组合的过程。

### 参数空间

矩阵乘法涉及的参数非常庞大：

```
参数：
  m_tile     ∈ {32, 64, 128, 256}
  n_tile     ∈ {32, 64, 128, 256}
  k_tile     ∈ {128, 256, 512, 1024}
  展开因子   ∈ {1, 2, 4, 8}
  循环顺序   ∈ {M-N-K, M-K-N, K-M-N, ...}   (6 种)
  prefetch   ∈ {0, 64, 128, 256} bytes
  向量宽度   ∈ {128, 256, 512} bits
```

仅上述参数就有 **约 18,000 个可能配置**，且每款 CPU 的最优组合各不相同。

### Tuning 方法

| 方法 | 做法 | 代表 |
|---|---|---|
| **静态经验式** | 根据 L1/L2/L3 容量和缓存行大小手算 tile | 早期 BLIS |
| **运行时探测** | 启动时在真实硬件上跑微基准，选最优参数 | OpenBLAS、BLIS |
| **编译期 JIT** | 已知硬件参数，编译时生成专用代码 | LibXSMM、JAX/XLA |
| **离线 Auto-Tuning** | 在目标硬件上搜索参数空间，保存配置文件 | ATLAS、TVM、Halide |
| **预调优表** | 对已知 CPU 型号提前调优，按 MIDR 查表 | ARM Compute Library |

### 实践中如何使用

**OpenBLAS / BLIS**：

首次调用时运行微型 benchmark，探测最优 tile 大小，在进程生命周期内缓存结果。

**ARM Compute Library**：

对每款支持的核心（Cortex-A510、A715、X925、Neoverse N1 等）预先调优并存储为调度表文件，运行时根据 CPU ID 寄存器（MIDR_EL1）直接选择对应配置，零启动开销。

**TVM / Halide**：

作为编译流水线的一部分，离线运行 Auto-Tuner（如 XGBoost 驱动的搜索），为目标硬件生成调度方案，嵌入到编译产物中。

### 常见调优优先级

实际调优中，参数的重要度并不均等：

1. **tile 大小** — 决定缓存驻留率，影响最大
2. **循环顺序** — 决定数据复用模式
3. **展开因子** — 减少循环开销，填充流水线
4. **prefetch 距离** — 隐藏主存延迟
5. **向量宽度选择** — 在混合精度时权衡精度和带宽

---

## 三者关系

```
Tiling  → 决定计算顺序和数据搬运模式（"怎么切"）
Tailing → 处理不整除的情况（"剩下怎么办"）
Tuning  → 寻找最优的 tiling 参数（"切多大"）
```

一个完整的矩阵乘法库 = **tiling 策略 + tailing 处理 + tuned 参数**。三者缺一不可。
